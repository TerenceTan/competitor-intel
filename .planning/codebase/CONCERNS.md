# Codebase Concerns

**Analysis Date:** 2026-05-04

## Tech Debt

**Thunderbit Social Scraper Integration (Broken):**
- Issue: Thunderbit AI extraction for Facebook, Instagram, and X (Twitter) is integrated but returns empty/null data consistently
- Files: `scrapers/social_scraper.py` (lines 47, 122-168), `src/app/(dashboard)/page.tsx` shows social data gaps
- Impact: Social media metrics incomplete; Dashboard displays "—" for all social platforms, making competitive social positioning analysis impossible. Users see incomplete data.
- Fix approach: Migrate to Meta Graph API for Facebook/Instagram (native auth) and Twitter API v2 for X. Sunset Thunderbit integration; maintain ScraperAPI fallback as interim secondary source.

**SQLite Scaling Limitations:**
- Issue: Dashboard is on SQLite; marketing-portal has already migrated to Postgres. Schema design with JSON blobs (`leverageJson`, `spreadJson`, etc.) limits query flexibility.
- Files: `src/db/schema.ts` (lines 1-158), `src/db/migrate.ts`, database is `data/competitor-intel.db`
- Impact: JSON blobs prevent efficient filtering/aggregation at query time. Large snapshots in `pricing_snapshots`, `reputation_snapshots` cause full-table scans. Scaling to multi-tenant or high-frequency updates will require migration.
- Fix approach: Plan Postgres migration following marketing-portal pattern. Break JSON blobs into normalized tables (e.g., `account_types`, `leverage_configs`). Schedule post-v1.4.2 as separate phase.

**Large Component Files (Complexity Risk):**
- Issue: Multiple page components exceed 650 lines, making refactoring and testing difficult
- Files:
  - `src/app/(dashboard)/competitors/[id]/page.tsx` (1018 lines)
  - `src/components/admin/competitor-form.tsx` (705 lines)
  - `src/app/(dashboard)/markets/[code]/page.tsx` (681 lines)
  - `src/app/(dashboard)/page.tsx` (658 lines)
- Impact: High cognitive load; hard to refactor; bugs in one section ripple. Testing individual sections difficult. Long load times if data queries are slow.
- Fix approach: Extract reusable sub-components (data fetching, chart rendering, form sections). Create separate hooks for repeated query patterns. No immediate blocking issue but refactor before adding major features.

**In-Memory Rate Limiting (Unbounded Map Growth):**
- Issue: Rate limiter Map in `src/middleware.ts` has opportunistic cleanup but no hard limit; could grow unbounded under extreme load
- Files: `src/middleware.ts` (lines 36-60), cleanup at line 47: `if (rateLimitBuckets.size > 1000)`
- Impact: Very low risk in single-instance (PM2) deployment, but if attack targets thousands of unique IPs/keys, memory leaks possible. No impact on v1.4.1 in current EC2 setup.
- Fix approach: Move rate limiting to Redis if scaling horizontally; for now, acceptable for single-host. Add logging if cleanup triggers.

**Logs Directory Not Gitignored (Potential PII):**
- Issue: `logs/` directory is in `.gitignore` but logs are generated locally without cleanup automation
- Files: `logs/` directory exists with `.gitkeep`, logs are generated by scrapers (e.g., `ai-analyzer.log`, `reputation-scraper.log`)
- Impact: Low—logs are already gitignored. However, no automated log rotation or compression. Old logs accumulate on disk. No PII in code, but depend on scrapers not leaking data.
- Fix approach: Not urgent. Add log rotation via cron/systemd on EC2. Verify no API keys are logged during production runs.

**Password Hashing in Two Places (Inconsistency):**
- Issue: Two implementations of password hashing—both correct, but duplication
- Files:
  - `src/middleware.ts` (lines 7-13): Uses Web Crypto API
  - `src/app/api/auth/login/route.ts` (lines 4-5): Uses Node.js `createHash`
  - `src/app/api/admin/run-scraper/route.ts` (lines 36-37): Uses Node.js `createHash`
- Impact: Low risk; both are correct SHA-256. But inconsistency could lead to bugs if one is changed. Three locations to update if algorithm changes.
- Fix approach: Extract to shared utility function `src/lib/crypto.ts` and use everywhere.

**API Discovery Function Has SSRF Mitigation But DNS Rebinding Risk:**
- Issue: `discoverCompetitorConfig()` in `src/app/(dashboard)/admin/actions.ts` (lines 130-159) blocks private IPs but doesn't re-check after DNS resolution
- Files: `src/app/(dashboard)/admin/actions.ts` (lines 162-175)
- Impact: Low—DNS rebinding attack would require admin to click on a crafted link, and the window is small. Protected by authentication layer.
- Fix approach: Not urgent. If discovery is heavily used by untrusted admins, add DNS resolution before fetch and re-validate IP.

## Known Bugs

**Thunderbit Social Extraction Returns Empty Data:**
- Symptoms: Social snapshots created but all follower counts, engagement metrics are null. Frontend displays "—" for all platforms.
- Files: `scrapers/social_scraper.py` (lines 125-168: `_thunderbit_extract()`)
- Trigger: Run `python scrapers/social_scraper.py` or scheduled scraper job
- Root cause: Thunderbit API schema format changed or API key invalid; logs show "empty extract" or "success=false" errors (check `logs/social-scraper.log`)
- Workaround: Comment out Thunderbit paths in `social_scraper.py` and rely on YouTube API + ScraperAPI fallbacks for Instagram/Facebook (limited data)
- Status: v1.4.1 known issue per context

**Reputation Scraper May Fail on Playwright Browser Launch:**
- Symptoms: Reputation data incomplete; `logs/reputation-scraper.log` shows browser timeout errors
- Files: `scrapers/reputation_scraper.py` (lines 68-79, 187+: uses `async_playwright()`)
- Trigger: Run on systems without Playwright dependencies or slow network
- Cause: Playwright requires chromium binary; curl_cffi optional but MyFXBook scraping skipped if missing
- Workaround: Install dependencies: `pip install playwright curl-cffi; playwright install chromium`
- Status: Not yet observed on EC2 but possible on local dev

**Account Types Extraction May Overwrite with Empty Data:**
- Symptoms: `pricing_snapshots` shows blank `account_types_json` on some competitors despite being present on website
- Files: `scrapers/pricing_scraper.py` (lines 113-160: guards with `if not accounts` but context-dependent)
- Trigger: Claude API extraction returns `{"accounts": []}` due to page structure variation
- Cause: Claude prompt doesn't always find account types on unusual page layouts
- Status: Partially mitigated by "never overwrite if empty" check; still loses data on first run if layout unfamiliar

## Security Considerations

**API Key Leakage in Logs (Partially Mitigated):**
- Risk: Scrapers pass API keys (ANTHROPIC_API_KEY, THUNDERBIT_API_KEY, YOUTUBE_API_KEY) via environment; if printed to logs, keys leak
- Files: `scrapers/social_scraper.py`, `scrapers/pricing_scraper.py`, `scrapers/reputation_scraper.py`, `scrapers/ai_analyzer.py`
- Current mitigation: Keys are passed to `requests.post()` headers, not logged in error messages; error messages log `response.text[:300]` which truncates body
- Recommendation: Add audit check to ensure no `api_key`, `key=`, `Authorization:` patterns appear in error messages. Use structured logging with redaction.
- Code review needed: Especially in exception handlers where error messages are built

**SSRF Prevention in Auto-Discovery (Adequate):**
- Risk: Admin auto-discovery could probe internal services (EC2 metadata endpoint, RDS, Redis on VPC)
- Current mitigation: `isPublicHttpUrl()` in `src/app/(dashboard)/admin/actions.ts` (lines 130-159) blocks RFC1918, loopback, link-local, IPv6 ULA
- Status: Good. DNS rebinding not fully blocked but attack surface is small (requires admin to enter URL).

**Password-Based Auth (Acceptable for Internal Tool):**
- Risk: No multi-factor auth, no OAuth
- Files: `src/app/api/auth/login/route.ts`, `src/middleware.ts`
- Current mitigation: Rate limiting (5 attempts per IP in 5-min window), constant-time comparison, timing-safe hash check
- Status: Adequate for internal-only dashboard. If exposed to internet, add MFA via next-auth v5 (already in dependencies).

**No Content Security Policy Violations (Good):**
- CSP header in `next.config.js` (line 19-31) is tight; no `dangerouslySetInnerHTML` patterns found in codebase
- No eval/Function constructor calls detected
- Status: Clean

**Rate Limiting on API Routes (Good):**
- `/api/v1/*` routes protected by middleware rate limit (60 req/min per key/IP)
- `/api/auth/login` protected by login rate limiter (5 attempts per IP in 5-min window)
- Status: Adequate for v1.4.1

**Database Access (SQLite — No SQL Injection Risk):**
- All queries use Drizzle ORM; no raw SQL concatenation found
- Python scrapers use parameterized queries via `db.execute(sql, params)`
- Status: No SQL injection risk detected

## Performance Bottlenecks

**Largest Page Components Have N+1 Query Patterns (Potential):**
- Problem: `src/app/(dashboard)/competitors/[id]/page.tsx` (1018 lines) fetches competitor + 9 parallel data streams (pricing, promo, social, reputation, etc.). If each snapshot requires a separate query, risk of slow page loads.
- Files: `src/app/(dashboard)/competitors/[id]/page.tsx` (lines 85-139 use Promise.all but each `.select().from().where().orderBy().limit(1)` is a separate query)
- Current state: Uses Promise.all so no sequential delay, but Drizzle ORM may not batch requests. Acceptable for single-competitor view.
- Improvement path: Profile with Drizzle logs enabled; if slow, consider denormalization or query batching. Not blocking v1.4.1.

**Market Filter Query on All Pages (Acceptable):**
- Problem: Every page queries with market filter, but SQLite full-table scan on JSON fields
- Files: `src/app/api/v1/trends/route.ts` (line 47: `eq(promoSnapshots.marketCode, marketFilter)`)
- Current state: Only ~10k rows in production so acceptable. Would degrade past 100k snapshots.
- Improvement path: Add composite index on `(competitor_id, market_code, snapshot_date)` after migration to Postgres

**Prometheus Metrics Missing (Observability):**
- Problem: No built-in metrics for API latency, database query times, scraper duration
- Impact: Hard to spot performance degradation in production
- Recommendation: Add instrumentation (OpenTelemetry or simple metrics endpoint) in next phase

## Fragile Areas

**Competitor Config Discovery (Admin Manual Fallback Needed):**
- Files: `src/app/(dashboard)/admin/actions.ts` (lines 246-350: `discoverCompetitorConfig()`)
- Why fragile: Regex-based link extraction for social URLs, pricing pages. If a broker changes URL structure (e.g., `/en/pricing` → `/pricing-en`), auto-discovery fails silently.
- Safe modification: Add logging of what patterns failed; document known link patterns per broker; add manual override fields in form UI.
- Test coverage: No tests for discovery function; relies on manual testing in dev

**Scraper Orchestration (run_all.py):**
- Files: `scrapers/run_all.py` (147 lines), `src/app/api/admin/run-scraper/route.ts`
- Why fragile: Sequential scraper execution; if one hangs, others are blocked. No scraper timeout enforced; process can accumulate zombie child processes.
- Safe modification: Add per-scraper timeout (30 min max), capture SIGTERM gracefully, add monitoring of scraper-runs table to alert on hung jobs
- Test coverage: Not testable; relies on manual runs

**JSON Schema in Database (Breaking Changes Risk):**
- Files: `src/db/schema.ts` all `*Json` columns, `src/lib/utils.ts` `safeParseJson()` (lines 95-102)
- Why fragile: Scrapers insert JSON that consumers expect in specific format. If Claude/Thunderbit format changes, parsing breaks silently (defaults to empty array `[]`)
- Safe modification: Add schema versioning; validate JSON on insert; log parse failures
- Test coverage: No unit tests for JSON parsing; relies on integration tests

## Scaling Limits

**SQLite Single Writer (Already Known):**
- Current capacity: ~50k competitor snapshots per month across all tables; dashboard handles it fine
- Limit: After ~1 year (500k snapshots), full-text search and aggregations slow. Lock contention if scrapers and web requests compete.
- Scaling path: Postgres migration (partially planned for marketing-portal). Estimated work: 2-3 sprints.

**Memory Usage of Rate Limiter Map:**
- Current capacity: 1000 IP/key entries with cleanup
- Limit: If deployed behind reverse proxy with many client IPs, map could grow to thousands before cleanup triggers
- Scaling path: Move to Redis if horizontal scaling planned

## Dependencies at Risk

- Risk: Dependency on beta version instead of stable v5 or v4
- Files: `package.json` (line 21)
- Impact: v5 API may change before release; future upgrade could break login flow
- Migration plan: Wait for v5 stable release (expected Q2 2026) and upgrade in next phase. Or downgrade to v4.x now if stability preferred.

- Files: `scrapers/reputation_scraper.py`, `scrapers/social_scraper.py`
- Impact: Slow `npm install` / `pip install` if accidentally required globally. Mitigated by keeping in `scrapers/` dir.

**Claude SDK (Anthropic):**
- Risk: Pricing scraper depends on Anthropic API; if service degraded, pricing extraction fails
- Files: `scrapers/pricing_scraper.py` (lines 60-140)

## Missing Critical Features

**Health Check Endpoint:**
- Problem: No `/health` or `/readiness` endpoint for monitoring. Can't tell if dashboard is up without hitting a page.
- Impact: EC2 load balancer (if any) can't detect app crashes. Manual intervention needed.
- Blocking: Not blocking, but should add before multi-instance deployment

**Structured Logging / Observability:**
- Problem: Logs via `console.log()` in Node, `print()` in Python. No trace IDs, no log aggregation setup
- Impact: Hard to correlate scraper failures with API requests; can't track request latency
- Blocking: Not blocking v1.4.1

**Audit Trail for Admin Actions:**
- Problem: No log of who added/edited competitors, ran scrapers, etc.
- Impact: Can't trace changes; security risk if admin credentials compromised
- Blocking: Not blocking; nice-to-have for compliance
