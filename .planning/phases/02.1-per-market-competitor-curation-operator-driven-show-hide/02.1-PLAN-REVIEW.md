- [MEDIUM] **`onConflictDoUpdate` with composite-PK target — no prior use in
  codebase.** Plan 02.1-04 Task 1 uses:
  ```typescript

    target: [competitorMarkets.competitorId, competitorMarkets.marketCode],

  ```
  The API exists in Drizzle 0.45.1 (confirmed via
  `node_modules/drizzle-orm/sqlite-core/query-builders/insert.js`) but this is

## Drizzle Composite-PK Upsert Compatibility

**Findings:**

- `package.json` confirms `drizzle-orm: ^0.45.1`.
- `node_modules/drizzle-orm/sqlite-core/query-builders/insert.js` exports
  `onConflictDoUpdate(config)` — the method exists.
- `node_modules/drizzle-orm/sqlite-core/primary-keys.js` exports
  `function primaryKey(...config)` — composite PK support confirmed.
- Zero existing uses of `onConflictDoUpdate` or composite `primaryKey` tables
  in `src/` (grep confirmed).

**Assessment for Plan 02.1-01 (composite PK in schema.ts):**
The Drizzle `primaryKey({ columns: [t.competitorId, t.marketCode] })` API is
supported and matches the documented signature. The executor must extend the
import on line 1 of `schema.ts` to add `primaryKey`. This is LOW runtime risk
(TypeScript will catch any import error immediately via `tsc --noEmit`).

**Assessment for Plan 02.1-04 (`onConflictDoUpdate` on composite PK target):**
The API exists but has no prior use in this codebase. The Drizzle insert
builder's `onConflictDoUpdate` method accepts an array for `target` (composite
conflict target), which maps to SQLite's `ON CONFLICT(col1, col2) DO UPDATE`.
This SHOULD work correctly with a composite primary key. However, since it is
untested in this project, the executor should:
1. Verify `tsc --noEmit` passes before committing.
2. If runtime behavior is unexpected (Drizzle generates malformed SQL), use
   the raw SQL fallback the plan documents: the recommended fallback is
   `INSERT OR REPLACE INTO competitor_markets (...) VALUES (?, ?, ?, ?, ?)` via
   `conn.execute(...)` or equivalently using Drizzle's `sql` tagged template.

**Recommendation:** The executor should attempt `onConflictDoUpdate` first
(Drizzle's typed API is preferable). If the TypeScript or runtime test fails,
fall back to the raw SQL path. The plan's fallback note is sufficient for an
experienced executor but could be more explicit for edge cases.
