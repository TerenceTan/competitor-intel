# Phase 2.1 — Plan Index

**Phase:** 02.1-per-market-competitor-curation-operator-driven-show-hide
**Planned:** 2026-05-14
**Goal:** Marketing managers see only the brokers that are actually competing in their market. Operator-curated per-market SHOW/HIDE via a new `competitor_markets` table, with default-safe fallback to Phase 2 behavior (show all) when the table is empty so the code can ship immediately and activate per market as marketing fills in the data.

```

  └── 02.1-01  competitor_markets table — Drizzle (src/db/schema.ts) + Python migration (scrapers/db_utils.py)
```

## Operator Setup Gates (carry into SUMMARYs)

Surface these in each plan's SUMMARY as the operator-action checklist:

1. **openpyxl install on EC2** (operator tool dep) — `.venv/bin/pip install openpyxl` before first import run. Per the build_competitor_xlsx.py docstring + Plan 02.1-02 docstring. NOT added to scrapers/requirements.txt.
2. **Marketing review round** — marketing fills the Decision column in `logs/PHASE_2_1_COMPETITOR_REVIEW.xlsx` (generated by `scrapers/admin/build_competitor_xlsx.py`). This is the gating step before the import script does anything useful. Async to engineering.
3. **First import run** — `.venv/bin/python scrapers/admin/import_market_decisions.py --dry-run` to preview, then `.venv/bin/python scrapers/admin/import_market_decisions.py` to commit. Operator-triggered, NOT cron-scheduled (D2.1-10).
4. **Ongoing edits via admin UI** — once seeded, marketing maintains per-market status via `/admin/competitors/<id>` directly. No engineering involvement needed (D2.1-08 long-term workflow).
5. **Spreadsheet vs admin UI: source-of-truth contract** — the spreadsheet is the seed source. Once marketing starts editing via the admin UI, re-running the import WITHOUT `--purge` is safe (it upserts, never deletes admin edits). Running WITH `--purge` will overwrite admin edits for any market_code present in the spreadsheet — document this in operator runbook.

These setup tasks are NOT plan tasks — they are operator pre-conditions for the feature to activate in production. Code ships safely either way; default behavior (empty table) preserves Phase 2 exactly.

## Risk Register

|-----------------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| R1: Marketing fills Decision column with values not in the dropdown (typos, leading/trailing whitespace)                          | Plan 02.1-02 normalizes via `.strip().lower()` before lookup against VALID_STATUSES; unknown values are SKIPPED with a stderr WARN message (not crashed). Re-import is safe.                                                                                                                                                            |
| R2: --purge accidentally wipes a market that has had admin-UI edits since the last import                                          | Plan 02.1-02 documents the upsert-only default; --purge is opt-in. Operator runbook (Phase 2.1 close) calls out the --purge semantics explicitly. Plan 02.1-04's admin route documents that the spreadsheet is the seed source.                                                                                                       |
| R3: CHECK constraint on status rejects a row mid-import → entire import rolls back                                                | Plan 02.1-02 wraps the entire UPSERT loop + audit row in a single transaction; if any row fails the CHECK, the transaction rolls back and the operator sees a clear stderr error. No partial-state DB.                                                                                                                                  |
| R4: Operator forgets to run the import after marketing review → dashboard shows Phase 2 behavior (looks "wrong")                  | D2.1-04 / D2.1-05 default-safe contract: empty table = Phase 2 behavior. Visually identical to pre-Phase-2.1 deploy. No broken UX. Operator can run the import any time without dashboard downtime.                                                                                                                                     |
| R5: SERP CSV for some markets missing (operator hasn't run serp_market_research.py for all 8)                                     | Plan 02.1-03's `readEmergingSerpRows` returns `[]` on missing file. No error UI, no broken page.                                                                                                                                                                                  |

| R8: Drizzle's `onConflictDoUpdate` doesn't support composite PKs on this Drizzle version                                          | Plan 02.1-04's <action> documents the fallback: raw `INSERT OR REPLACE` via `db.run(sql\`...\`)` template. Either is correct; the verification gate is the audit-row shape, not the SQL flavor.                                                                                                                                       |

## Phase Exit Criteria

Phase 2.1 is COMPLETE when:

1. All 4 plans (02.1-01..02.1-04) have shipped + SUMMARY.md files.
2. `competitor_markets` table exists in the production DB (verifiable via `sqlite3 data/competitor-intel.db ".schema competitor_markets"`).
3. `scrapers/admin/import_market_decisions.py --help` runs cleanly on EC2.
4. `/admin/competitors/pepperstone` renders without error against the production DB (8 markets, all "Not curated" until first save).
5. `/admin/data-health` shows the "Per-market curation: N of 8 markets seeded" tile (initial N = 0; updates as marketing seeds).
6. `/markets/sg` (or any APAC v1 code) loads identically to Phase 2 behavior when `competitor_markets` is empty (default-safe contract verified).
7. ROADMAP Phase 2.1 marked complete; STATE.md updated.

After Phase 2.1 lands, the operator can:
- Run the import script when marketing completes the first xlsx review (criterion #5 satisfied for SG + VN if marketing fills those rows).
- Edit individual rows via `/admin/competitors/<id>` for ongoing maintenance.
- Verify via `/admin/data-health` that the markets-seeded count climbs as reviews finish.

The dashboard activates per-market filtering automatically on the next page render of `/markets/<code>` after any write — no env var flip, no redeploy.
