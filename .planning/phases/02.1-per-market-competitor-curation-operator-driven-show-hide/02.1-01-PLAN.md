<objective>
Create the `competitor_markets` table that powers Phase 2.1's per-market SHOW/HIDE curation. This is the schema foundation that everything downstream depends on:
- Plan 02.1-02 (operator import script) writes rows here.
- Plan 02.1-03 (UI filter + Emerging rail on `/markets/[code]`) reads rows here.
- Plan 02.1-04 (admin per-market status editor) reads + writes rows here.

Per D2.1-01, the table is `(competitor_id, market_code, status, notes, updated_at)` with composite primary key `(competitor_id, market_code)`, a CHECK constraint restricting `status` to `'active' | 'planned' | 'withdrawn' | 'emerging'`, and a foreign key from `competitor_id` to `competitors(id)`. SQLite has no native enum type — the CHECK constraint is how we enforce the value set.

Per D2.1-02, BOTH the Drizzle schema (`src/db/schema.ts`) AND the Python migration block (`scrapers/db_utils.py get_db()`) are updated in the same plan so dashboard reads and operator-side seed-import writes can't drift. Per D2.1-03, the migration is additive only — no ALTER TABLE on existing tables, no FK changes, just a new table. Matches the Phase 1 / Plan 01-01 idempotent CREATE TABLE IF NOT EXISTS pattern.

Per D2.1-04 / D2.1-05, the table being empty MUST behave identically to Phase 2 (show all competitors). This is enforced at the query level by Plan 02.1-03; this plan just creates the empty table.

**Out of scope for this plan:**
- The audit-trail columns (`approved_by`, `approved_at`) — deferred to a hypothetical Phase 2.2 per CONTEXT.md `<domain>` "Out of scope" + the Claude's Discretion note about `last_reviewed_at`.
- Any data inserts. The table ships empty. Seeding happens via Plan 02.1-02's import script (operator-triggered).
- Any UI filter logic. Plan 02.1-03 owns that.
- Any admin write actions. Plan 02.1-04 owns that.
- Cron-scheduled imports. Per D2.1-10, the import is operator-triggered.

Purpose: Unblock the entire Phase 2.1 wave-2 fanout. This is a small, well-bounded change — one CREATE TABLE block in Python + one `sqliteTable` export in TypeScript — but it has to land first because the rest of the phase reads/writes through it.

Output: A new `competitor_markets` table in `data/competitor-intel.db` (created on first server boot or scraper run after deploy), and a typed `competitorMarkets` Drizzle export for the dashboard side. No data migration; existing rows in other tables are untouched.

The Drizzle SQLite-core API supports composite primary keys via the second argument of `sqliteTable`:
```typescript
sqliteTable("name", { /* columns */ }, (t) => ({
  pk: primaryKey({ columns: [t.competitorId, t.marketCode] }),
}));
```

SQLite supports `CHECK (col IN (...))` for enum-style constraints. Example:
```sql
status TEXT NOT NULL CHECK (status IN ('active','planned','withdrawn','emerging'))
```
