# Phase 02.1 Plan 01: Schema for competitor_markets Summary

Add the `competitor_markets` table (Drizzle + SQLite migration block) that powers Phase 2.1's per-market SHOW/HIDE curation — composite PK on `(competitor_id, market_code)`, FK on `competitor_id → competitors(id)`, CHECK constraint restricting `status` to `'active' | 'planned' | 'withdrawn' | 'emerging'`, and an idx_competitor_markets_market_status composite index for the active-filter access pattern.

## What Shipped

Two file changes, additive only — no existing table was touched.

### `scrapers/db_utils.py`

Appended a new migration block in `get_db()` between the existing `share_of_search_snapshots` block (line 199) and the Pepperstone `INSERT OR IGNORE` block (line 202):

```sql
CREATE TABLE IF NOT EXISTS competitor_markets (
    competitor_id TEXT NOT NULL REFERENCES competitors(id),
    market_code TEXT NOT NULL,
    status TEXT NOT NULL CHECK (status IN ('active','planned','withdrawn','emerging')),
    notes TEXT,
    updated_at TEXT NOT NULL,
    PRIMARY KEY (competitor_id, market_code)
)
```

Plus the composite index:

```sql
CREATE INDEX IF NOT EXISTS idx_competitor_markets_market_status
ON competitor_markets (market_code, status)
```

The migration is idempotent (`CREATE TABLE IF NOT EXISTS` + `CREATE INDEX IF NOT EXISTS`) — safe to run on every connection open, matching the Phase 1 / Plan 01-01 pattern. On production EC2 it will materialize the table on first scraper run after deploy.

### `src/db/schema.ts`

Two changes:

1. Extended the drizzle-orm/sqlite-core import to include `primaryKey` (alongside the existing `sqliteTable`, `text`, `integer`, `real`).
2. Appended `competitorMarkets` table export at the end of the file (below `scraperRuns`):

```typescript
export const competitorMarkets = sqliteTable(
  "competitor_markets",
  {
    competitorId: text("competitor_id").notNull().references(() => competitors.id),
    marketCode: text("market_code").notNull(),
    status: text("status").notNull(),
    notes: text("notes"),
    updatedAt: text("updated_at").notNull(),
  },
  (t) => ({
    pk: primaryKey({ columns: [t.competitorId, t.marketCode] }),
  }),
);
```

`status` is typed as plain `text()` — the CHECK constraint enforces the enum at the DB layer (no native SQLite enum), and consumers (Plan 02.1-03, 02.1-04) narrow it with `as 'active' | 'planned' | 'withdrawn' | 'emerging'` at use sites. This matches the Phase 1 convention (`apifyRunLogs.status` is also a plain text column with values constrained elsewhere).
