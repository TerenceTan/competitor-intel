# Phase 02.1 Plan 04: Admin Per-Market Status Editor + Data-Health Markets-Seeded Tile Summary

Ship the admin UI for editing per-market competitor status ŌĆö operators set a 4-value status (active/planned/withdrawn/emerging) and free-text notes per (competitor, market) tuple, with every write audited to change_events and the dashboard immediately reflecting changes via revalidatePath. Closes ROADMAP Phase 2.1 success criterion #4 (admin UI without redeploys) and adds a data-health tile (D2.1-12) showing N of 8 markets seeded.

## What Shipped

### Server actions (src/app/(dashboard)/admin/actions.ts)

- **`upsertCompetitorMarket(competitorId, marketCode, status, notes)`** ŌĆö Validates competitorId (FK pre-flight), marketCode against PRIORITY_MARKETS, status against the 4-value enum. Reads prior status, then UPSERTs via Drizzle's `onConflictDoUpdate` on the composite PK. Writes a `change_events` audit row with `field_name='market_curation_edit'`, `domain='admin'`, `severity='low'`, `oldValue=<prior>`, `newValue=<new>`. Calls `revalidatePath` on /admin/competitors/<id>, /markets/<code>, /admin/data-health. Returns `{ success: true }` or `{ error: string }`.

- **`clearCompetitorMarket(competitorId, marketCode)`** ŌĆö Validates inputs. Reads prior status; if no prior row, returns success as idempotent no-op (no audit row written). Otherwise DELETEs the row, writes audit row with `newValue=null`. Same revalidatePath fan-out.

- **Local validation helpers** `isCuratedMarketCode` + `isCuratedStatus` ŌĆö kept private (not exported) to keep the trust boundary tight at the server-action surface.

### Client component (src/components/admin/per-market-status-table.tsx)

- Renders one row per `PRIORITY_MARKETS` entry (8 rows) with flag + market name + code on the left, status dropdown + notes input in the middle, Save/Clear button on the right.
- Initial state hydrated from `existingRows` prop; rows without an existing competitor_markets entry render "ŌĆö Not curated ŌĆö" with empty notes and a disabled Save button.
- Per-row dirty + saving state; Save button enabled only when dirty + not currently saving.
- On Save: calls `upsertCompetitorMarket` (status non-empty) or `clearCompetitorMarket` (status = "" / Not curated). Sonner toast on success/error. `router.refresh()` wrapped in `useTransition` to re-pull the server-component data and reset dirty state.

### Admin route (src/app/(dashboard)/admin/competitors/[id]/page.tsx)

- Server component, `force-dynamic`. Validates the id param shape (lowercase alphanumeric + hyphens, Ōēż50 chars ŌĆö mirrors `validateId` in actions.ts:40-44).
- Two parallel Drizzle queries via `Promise.all`: one for the competitor row, one for all matching competitor_markets rows. `notFound()` for invalid id shape or unknown competitor.
- Renders breadcrumb (ŌåÉ Admin / Competitors / <name>), header with name + id (mono) + website link (with noopener + noreferrer) + tier, then a `<section>` with an explainer paragraph + the `PerMarketStatusTable` client component.

### Data-health tile (src/app/(dashboard)/admin/data-health/page.tsx)

- Added `competitorMarkets` to the existing schema import.
- Extended the `Promise.all` from 4 ŌåÆ 5 parallel queries; the new query is a `SELECT marketCode FROM competitor_markets GROUP BY marketCode` (counts distinct seeded markets).
- New `seededMarkets = seededRows.length` derivation.
- New `<section>` between the existing Apify Cost tile and the Scrapers table ŌĆö renders "N of 8 markets seeded" with matching `rounded-xl border border-gray-200 bg-white p-6` styling. When N<8, a helper paragraph nudges operators to either the bulk import script or `/admin/competitors/<id>`.

## How To Use

**Direct route (no link from /admin yet ŌĆö per plan default, link wiring deferred):**
- Navigate to `/admin/competitors/<id>` where `<id>` is a known competitor id (e.g. `/admin/competitors/pepperstone`, `/admin/competitors/ic-markets`).
- For each of the 8 APAC v1 markets: pick a status from the dropdown, optionally fill in marketing rationale in Notes, click Save.
- To clear curation for a market: set status to "ŌĆö Not curated ŌĆö" and click Clear.

**Audit trail:**
```sql
sqlite3 data/competitor-intel.db "SELECT detected_at, competitor_id, market_code, old_value, new_value FROM change_events WHERE field_name='market_curation_edit' ORDER BY detected_at DESC LIMIT 20"
```

**Seeded-markets status:**
- Visit `/admin/data-health`. The "Per-market curation" tile (between Apify Cost and Scrapers table) shows current seeded count.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Renamed local `isMarketCode` helper to `isCuratedMarketCode`**
- **Found during:** Task 1 (server actions implementation)
- **Issue:** The plan suggested naming the local validation helper `isMarketCode`, but `src/lib/markets.ts` already exports a public function with that exact name (different signature ŌĆö accepts `string | null | undefined`). Importing one + defining the other locally is a TypeScript error.
- **Fix:** Renamed the local helper to `isCuratedMarketCode` (signals it validates against the curated PRIORITY_MARKETS list specifically). Same call sites, same logic, no behavioral change.
- **Files modified:** `src/app/(dashboard)/admin/actions.ts`
- **Commit:** `ebb837e`

**2. [Rule 3 - Blocking] tsc invocation via parent node_modules**
- **Found during:** Task 1 verification
- **Issue:** This worktree shares parent `node_modules/` symlinks but `npx tsc` and `node node_modules/typescript/lib/tsc.js` both fail because the symlink resolves relative to the worktree. Per the `<parallel_execution>` block in the executor preamble this is a known limitation.
- **Fix:** Invoked tsc directly via the parent repo path (`node $PARENT_REPO/node_modules/typescript/lib/tsc.js --noEmit -p tsconfig.json`). Verification mechanic only ŌĆö no code change.

### Build retry (transient, not a deviation)

The first `npm run build` invocation hit `SqliteError: database is locked` (SQLITE_BUSY) during Next.js's page-data collection step on `/api/changes` and `/api/v1/trends`. This is environmental ŌĆö the parent repo and other worktree builds had open WAL handles on the same SQLite file at the same time. Re-running `npm run build` produced a clean "Compiled successfully in 1018ms". Not a code defect, just multi-worktree concurrency on the local DB.
