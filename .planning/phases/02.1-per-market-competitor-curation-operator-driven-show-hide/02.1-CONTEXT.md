<domain>
## Phase Boundary

Phase 2 fanned scrapers out to 8 APAC markets but every `/markets/<code>` view
shows all 15 competitors regardless of whether they're actually active in
that market — too noisy. Phase 2.1 introduces operator-curated per-market
SHOW/HIDE driven by:

1. **SERP signal** — Apify google-search-scraper, 15 queries/market, own-brand
   filtered, ccTLD-aware domain matcher
2. **5-signal validator** — local URL, language, payment, app store, WikiFX
3. **Operator approval** — marketing-team review of consolidated Excel matrix

Phase 2.1 ships the infrastructure (`competitor_markets` table, filter, admin
UI, Emerging rail) BEFORE marketing review lands — design defaults to "show all"
when the table is empty so behavior is unchanged from Phase 2 until the seed
data arrives. Marketing review fills the table async via spreadsheet → seed
import OR the admin UI directly.

**In scope:**
- `competitor_markets` table (Drizzle + Python schema) with `status` enum
- `/markets/<code>` filter joining `competitor_markets` for `status='active'`
- "Emerging" rail at bottom of `/markets/<code>` showing STRONG-SERP competitors
  not yet in `competitor_markets` as `active`
- Admin endpoint + UI to edit per-market status
- Seed importer that reads marketing-approved Excel → INSERTs rows
- Default-to-current-behavior when the table is empty

**Out of scope:**
- Marketing team's actual review (operator-side; happens async via the
  shipped review xlsx in `logs/PHASE_2_1_COMPETITOR_REVIEW.xlsx`)
- Schema migration for additional fields (e.g., `approved_by`, `approved_at`
  audit trail) — defer to Phase 2.2 if needed
- Per-market handle research in `scrapers/config.py` — Option B from Phase 2
  Open Q1; still deferred
</domain>

<decisions>
## Implementation Decisions

### Schema and data model
- **D2.1-01 (LOCKED):** New table `competitor_markets(competitor_id TEXT,
  market_code TEXT, status TEXT, notes TEXT, updated_at TEXT)`.
  Composite primary key `(competitor_id, market_code)`. Foreign key on
  `competitor_id → competitors(id)`. `status` is a TEXT enum (SQLite has
  no native enum) validated by CHECK constraint to one of:
  `'active' | 'planned' | 'withdrawn' | 'emerging'`. Absent row = "not
  curated → hide".
- **D2.1-02 (LOCKED):** Both Drizzle schema (`src/db/schema.ts`) AND Python
  side (`scrapers/db_utils.py`'s migration block) updated in the same plan
  so dashboard reads and operator-side seed-import writes don't drift.
- **D2.1-03 (LOCKED):** Migration is additive only (CREATE TABLE IF NOT
  EXISTS); existing `social_snapshots`, `apify_run_logs`, `competitors`
  rows are untouched. Matches the Phase 1 / Plan 01-01 idempotent pattern.

### Default behavior (Phase 2 preserved)
- **D2.1-04 (LOCKED):** When `competitor_markets` is EMPTY for a given market,
  `/markets/<code>` falls back to "show all competitors" (current Phase 2
  behavior). This ships the code safely before marketing data lands — no
  blank pages, no broken UX.
- **D2.1-05 (LOCKED):** Feature is "on" automatically once the table has any
  rows for the market. No env var or config toggle — the data itself drives
  the behavior.

### UI surfaces
- **D2.1-06 (LOCKED):** `/markets/<code>` Digital Presence section (shipped
  in Plan 02-04) extends to filter the `peerRanking` resolver by
  `competitor_markets.status='active'` for the requested market — joining on
  the existing latest-per-(competitor,platform,market) dedup logic from
  Phase 2. Empty `competitor_markets` → no filter applied.
- **D2.1-07 (LOCKED):** "Emerging Competitors" rail rendered below the main
  Digital Presence grid. Pulls from the SERP CSV (already produced by
  `serp_market_research.py` and stored at `logs/serp_research_<market>.csv`).
  Shows competitors with `queries_appeared >= 2 AND best_rank <= 10` (the
  STRONG threshold) but no `active` row in `competitor_markets`. Visual
  treatment: smaller cards, "Watching" badge, no detailed metrics — just
  name + SERP rank summary + a link to the broker site.
- **D2.1-08 (LOCKED):** Admin UI extension: `/admin/competitors/<id>` (or
  similar — operator decides exact route) adds a per-market status table.
  Operator can edit status per (competitor, market) without leaving the
  dashboard. Reuses the existing CompetitorTable component pattern from
  Plan 01-05.

### Seed import
- **D2.1-09 (LOCKED):** Operator script `scrapers/admin/import_market_decisions.py`
  reads `logs/PHASE_2_1_COMPETITOR_REVIEW.xlsx` (the marketing-approved
  workbook from `build_competitor_xlsx.py`) and emits INSERT/REPLACE
  statements for `competitor_markets`. Re-runnable: full upsert semantics
  on `(competitor_id, market_code)`, blanks/empty Decision values skipped.
- **D2.1-10 (LOCKED):** Import is operator-triggered, NOT automated. The
  marketing team's Decision column is the source of truth; engineering
  imports it after each review round. Auditable: every import logs a
  `change_events` row of `field_name='market_curation_imported'` with
  the row count and timestamp.

### Trust UX continuity
- **D2.1-11 (LOCKED):** All existing Phase 1 trust patterns preserved:
  `<EmptyState reason="scraper-failed">` still applies when a (competitor,
  market) has no social_snapshots row AND a recent scraper_zero_results
  event. The Phase 2.1 filter ADDS a hide-by-status layer on top of that —
  these are independent and don't collide.
- **D2.1-12 (LOCKED):** Admin Data Health page (extended in Plan 02-05)
  gets a small additional row: "Markets with curated competitor lists"
  showing `N of 8 markets seeded` for at-a-glance status of the review.

### Cost
- **D2.1-13 (LOCKED):** Pure code change — zero new Apify or external
  service cost. Reuses Phase 2 data flow. The SERP CSVs for the Emerging
  rail are already produced weekly by the existing operator pipeline.

### Claude's Discretion
- Exact SQL query shape for the `/markets/<code>` filter join — Drizzle
  `leftJoin` vs `innerJoin` with a fallback subquery; planner decides
  based on what's idiomatic in the existing codebase.
- Whether the Emerging rail dedupes against `status='emerging'` (already
  curated as emerging) or shows duplicates — recommend dedupe.
- Admin UI exact layout (single table vs per-market tabs) — planner
  decides based on existing CompetitorTable patterns.
- Whether to add a "Last reviewed at" timestamp column to
  `competitor_markets` now or defer to Phase 2.2 — recommend defer
  unless it's trivial.

</decisions>

<open_questions>
## Open Questions

None blocking — all design decisions are locked in `<decisions>`. Two
items the planner may want to clarify in the plans themselves:

1. **Q1: What does the admin UI look like for editing 15 × 8 = 120 status
   cells?** Per the discretion section, planner decides between a single
   table (15 rows × 8 cols + one dropdown per cell) vs a per-market tab
   layout. Recommend the per-market tab — easier mental model and matches
   the existing /markets/<code> mental model.

2. **Q2: When the marketing-team .xlsx is re-imported, do we replace ALL
   decisions for a market or only update the ones explicitly changed?**
   Per D2.1-09 we do full upsert on `(competitor_id, market_code)` —
   meaning a row removed from the spreadsheet would NOT be deleted from
   the DB (only blanks are skipped). If marketing wants "remove a row,
   broker disappears from the market" semantics, that's a different
   import contract. Recommend documenting the upsert-only semantics
   clearly in the import script's docstring + a `--purge` flag for
   destructive re-imports if needed.
</open_questions>
