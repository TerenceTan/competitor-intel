<objective>
Code review WR-05 caught a misleading comment in `scrapers/run_all.py`: the header asserts "child subprocesses install their own redaction" but in reality only 2 of 9 scripts in `SCRIPTS` actually do (`apify_social.py` and `calibration/validate_extraction.py`). The other 7 (`pricing_scraper.py`, `account_types_scraper.py`, `promo_scraper.py`, `social_scraper.py`, `reputation_scraper.py`, `wikifx_scraper.py`, `news_scraper.py`, `ai_analyzer.py`) use `print()` exclusively and bypass the `logging`-based redaction filter entirely.

Three available options per gap_closure_instructions:
- **(a)** Update the comment to reflect reality.
- **(b)** Add `install_redaction()` to all 9 scrapers' module preambles.
- **(c)** Move redaction to the parent `run_all.py` subprocess output capture.

This plan picks **(a)** — comment-only — for the following reasons grounded in the threat model anchored in `scrapers/log_redaction.py`'s April 2026 EC2 incident docstring:

- **(b) is out-of-phase scope.** Adding `install_redaction()` to 7 print-only scrapers does not actually protect them — `print()` writes directly to `sys.stdout` and bypasses every `logging.Filter`. The fix would have to migrate `print()` → `logging.info()` across the entire scraper fleet, which is a 7-file refactor that touches every scraper's hot path. That refactor risks breaking subprocess output capture in `run_all.py` (which currently parses stdout directly) and is not justified by Phase 1's scope. Phase 2 will already touch `social_scraper.py` for IG/X — that's the right moment to migrate it. The other 6 scrapers can be migrated incrementally as they're touched in Phases 2–5.
- **(c) is operationally risky.** Moving redaction to the parent `run_all.py` subprocess output capture would mean wrapping `result.stdout` and `result.stderr` in a redaction pass before writing to `logs/<scraper>.log`. This works but introduces a new failure mode (a redaction bug in the parent silently corrupts every scraper's log, not just one), and would still NOT protect the live stdout stream that operators tail via `journalctl` or SSH `tail -f`. The defense-in-depth value is real but the implementation cost vs benefit at Phase 1 doesn't justify the risk window.
- **(a) is the honest, minimum-risk move.** It tells the truth about the current state, names the future migration path, and references the threat model anchor (the April 2026 EC2 incident) so future readers understand WHY the partial coverage is acceptable for now. It does not change any executable code — zero regression risk on the cron pipeline.

**This plan is NOT a verification gap closure** (`closes_gaps: []`). It is a code-review defect fix bundled into the gap closure run because the modification is small, isolated, and adjacent to the other Phase 1 hardening work. The verification report did not flag this as an SC failure; the code review (WR-05) flagged it as a misleading comment that would mislead the next maintainer.
