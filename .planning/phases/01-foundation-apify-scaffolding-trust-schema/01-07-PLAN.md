<objective>
Wire `<EmptyState reason="scraper-failed">` into the Digital Presence tab of the competitor detail page so a recent zero-result Apify run renders as a visually distinct red empty state instead of the silent plain-text "N/A — Data unavailable" fallback.

The EmptyState variant exists (01-05). The data exists (apify_social.py writes a `change_events` row of type `scraper_zero_results` whenever an actor returns zero items, per 01-03). The wiring between them does not exist yet — gap 2 (SC2) from `01-VERIFICATION.md`.

This plan adds a per-platform "did the scraper fail recently?" lookup to the existing parallel data fetch in `src/app/(dashboard)/competitors/[id]/page.tsx` and branches the per-platform social card on it. Independent of Phase 1 operator follow-ups (an Apify run is not required for this plan to ship — the wiring becomes verifiable as soon as a `change_events` row of type `scraper_zero_results` exists for any competitor/platform).

Purpose: Close the silent-success gap. When the FB scraper returns zero items, marketing managers must see "scraper failed" (red AlertOctagon, distinct from a competitor with simply no FB activity) so they can route to triage instead of misreading the silence as the competitor going dormant.

Output: One modified file (`src/app/(dashboard)/competitors/[id]/page.tsx`) with one new Drizzle query in the existing `Promise.all` block, one Set built from the result, and one branched render in the per-platform social card.
</objective>
