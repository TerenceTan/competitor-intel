<objective>
Two surgical fixes that together address SC5's REAL bug (per code review WR-02) and reconcile the ROADMAP success criterion with the deferred-by-design data-collection step (per D-21).

**WR-02 — calibration validator sends market code as broker_name into the production prompt.** Line 221–226 of `validate_extraction.py` sends `item.get("market", "unknown")` (e.g. "TH") as the `broker_name` argument to `extract_promos_from_text` (which interpolates it into the prompt as `Broker: TH`). This biases extraction, makes per-language accuracy numbers unreliable, and would silently undermine any Phase 3 prompt-iteration decision based on this validator. Fix: the JSONL schema gains a `broker_name` field; the validator reads it; example rows carry placeholder values.

**ROADMAP SC5 reconciliation.** The roadmap success criterion as written ("a 20-30-item hand-labeled calibration set per non-English language exists in the repo") cannot be satisfied by code alone — hand-labeling is operator action. D-21 explicitly deferred data collection. Phase 1 delivered the validator + skeleton JSONL; the data-collection step belongs to Phase 3 prompt-iteration prep (or earlier operator action). Adding a brief reconciliation note to ROADMAP makes the deferral explicit so the next verifier doesn't re-flag it as a Phase 1 gap.

The actual hand-labeling of 100–150 promo snippets across 5 languages is **NOT** in scope for this plan — it's operator action with no programmatic substitute (per D-21 and gap_closure_instructions). This plan ships the bug fix and the ROADMAP reconciliation; the data-collection step continues to live with the operator.

Purpose: Make the calibration validator trustworthy the moment real data lands, and align the ROADMAP narrative with D-21 so the SC5 status is honest going forward.

Output: 3 modified files (Python validator, JSONL data file, roadmap markdown).
