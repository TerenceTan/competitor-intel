<objective>
Three defects, all in-phase, in two adjacent code paths — fix in one plan because they share the apify_social.py file and the Data Health correctness story:

1. **WR-01 / Gap SC3 — Data Health zero-result count is permanently broken.** The page does `z.actorId.includes(s.name) || z.actorId.includes(s.dbName)` where `s.name="apify-social"` / `s.dbName="apify_social"`, but the stored `actor_id` is `"apify/facebook-posts-scraper"`. Neither substring matches, so the column always shows 0. Fix: introduce an explicit `ACTOR_TO_SCRAPER` map in `src/lib/constants.ts` (single source of truth, lives next to the `SCRAPERS` constant it complements), and look up by equality.

2. **WR-03 — apify_social.py leaks two SQLite connections per run.** Two separate `conn = get_db()` calls (lines 183 and 275), neither closed. Harmless on short cron processes but masks correctness issues during interactive testing and is a bad pattern to leave as the canonical Apify boilerplate (other Phase 2 actors will copy this file). Fix: wrap both with `contextlib.closing()`.

3. **WR-04 — apify_social.py confidence rule is degenerate.** `confidence = "high" if follower_count and posts_last_7d is not None else "medium"` — `sum()` always returns an int ≥0, never None, so the second clause is always True and the rule reduces to `"high" if follower_count else "medium"`. Misleading: a run that returns 50 items but all timestamps fail to parse will report `posts_last_7d=0` AND `confidence="high"`. Fix: change to `posts_last_7d > 0` to match the docstring contract.

Purpose: Make the zero-result column on `/admin/data-health` deliver its stated SC3 value (currently it lies and shows 0 even when the database has empty-status rows). Tighten the Apify scraper boilerplate that Phase 2 will copy 8× across markets — fix the connection and confidence bugs before they propagate.

Output: 3 modified files (TS constants, TS page, Python scraper). All edits are surgical — no file is rewritten.
</objective>
