<objective>
Extend the operator Data Health page so per-market scraper failures are visible at a glance. Today the page shows total zero-result counts per scraper (e.g., "Apify Social Scraper: 3"). After Phase 2's per-market fanout, an operator needs to know **which markets are failing** — is it Singapore-wide? Just Vietnam? — so they can investigate proxy / geo / handle issues.

Per RESEARCH.md §7 the minimal change is a **second parallel Drizzle query** grouping by `(actorId, marketCode)` and rendering the breakdown as a compact badge string after the existing total count. Single-table layout is preserved (no new columns; do not break the page out into 8 per-market columns — that would harm scannability for the 80% case where most markets are healthy).

Specifically:

1. Add a 4th entry to the existing `Promise.all([...])` block: same `apify_run_logs` WHERE clause as the current zero-counts query (status='empty' + 7-day window) but with `GROUP BY (actor_id, market_code)`.

2. In the table cell rendering loop, look up the new breakdown rows for each SCRAPERS entry via the existing `ACTOR_TO_SCRAPER` equality match. Build a small inline badge string like `(sg: 1, my: 2)` ordered by market code alphabetically.

3. Skip the badge string entirely when the breakdown is empty OR contains only 'global' (the Phase 1 single-market state). Operators on the Phase 1 free tier should see the page identical to today.

**Out of scope:**
- No new columns / no row-per-market layout (RESEARCH.md §7 explicit guidance).
- No styling overhaul — reuse the existing amber `<span>` for non-zero totals.
- No new components in `src/components/`.
- No fix to the underlying scraper (that's Plan 02-03's responsibility); this plan only exposes diagnostics.
- No filtering UI / market selector on the Data Health page itself (admin pages are deliberately spartan).

Purpose: Make per-market scraper failures legible. Without this, operators would have to drop into the SQLite shell to figure out which market is misbehaving — friction the Phase 1 trust-UX work was specifically designed to eliminate.

Output: A 4-query Data Health page where the existing total counts are decorated with a small per-market breakdown when relevant.
</objective>
