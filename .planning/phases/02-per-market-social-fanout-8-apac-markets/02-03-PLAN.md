<objective>
Refactor `scrapers/apify_social.py` from "global-only single run" to "feature-flagged per-market loop" by threading `market_code` through every actor call, every DB write, and every `change_events` row. The refactor is a **parameter-threading exercise** (RESEARCH.md §1 Pattern 2) — not a rewrite. No new actors, no new schema, no new cron entry, no new external deps.

Specifically:

1. Add `_proxy_config(market_code: str) -> dict` helper near the top of the module (RESEARCH.md Pattern 1+2).
2. Add `TARGET_MARKETS = parse_target_markets(os.environ.get("APIFY_MARKETS_ENABLED"))` near the start of `run()`.
3. Change `run_facebook`, `run_instagram`, `run_x` signatures to accept `market_code: str`.
4. Inject `_proxy_config(market_code)` into all 5 actor call `run_input` dicts (FB pages, FB posts, IG profile, IG posts, X).
5. Replace every `DEFAULT_MARKET_CODE` literal inside the three `run_*` functions with the new `market_code` parameter — in `social_snapshots`, `apify_run_logs`, and `change_events` INSERTs.
6. Wrap the existing `run_facebook/run_instagram/run_x` calls inside `run()` in a `for market_code in TARGET_MARKETS:` loop (RESEARCH.md Pattern 2, lines 99-104).
7. Extend `scrapers/test_apify_social_market.py` (from Plan 02-02) with:
   - `TestProxyConfig` — unit tests for `_proxy_config("global")` and `_proxy_config("sg")` per RESEARCH.md Pitfall 1 (case normalization to uppercase).
   - One mock-Apify integration test that monkeypatches `ApifyClient.actor(...).call(...)` to return a deterministic fixture and asserts that calling `run_facebook(client, conn, run_id, snapshot_date, [('comp-a','test-slug')], market_code='my')` writes a `social_snapshots` row with `market_code='my'`. Use an in-memory sqlite DB (`sqlite3.connect(':memory:')` + apply the relevant CREATE TABLE statements from `db_utils.py`) — do NOT touch the production DB.

**This plan deliberately keeps changes additive and reversible:** `DEFAULT_MARKET_CODE` constant stays in the module (as the documented fallback), the cost caps stay unchanged, the zero-result guard stays unchanged, and the confidence rule stays unchanged. Only the threading of `market_code` and the loop are new.

**Critical Phase 1 invariants to preserve (per RESEARCH §security_domain and Phase 1 STATE.md):**
- `install_redaction()` BEFORE `from apify_client import ApifyClient` (April 2026 EC2 incident anchor; D-12 / INFRA-03).
- `with closing(get_db()) as conn:` block must wrap the entire market loop (one DB connection for all 8 market × 3 platform iterations — do NOT open/close per iteration).
- Per-actor `max_total_charge_usd` caps unchanged (D2-09 / D-06).
- Zero-result branch still writes a `change_events scraper_zero_results` row AND skips the `social_snapshots` INSERT (SOCIAL-04 carry-forward from Phase 1).
- `apify_run_logs` row is still INSERTed in the finally block on every iteration (success/empty/failure).
- ACTOR_BUILD pin (`0.0.293` for FB posts; the FB pages actor has no explicit build pin in Phase 1 — leave it alone unless RESEARCH says otherwise; it doesn't).
- Confidence rule: `high` IFF `follower_count > 0 AND posts_last_7d > 0`, else `medium`. Unchanged.

**Operator gate (per CONTEXT.md D2-08 and STATE.md ceiling concern):** This refactor is SAFE TO MERGE on the free tier because `APIFY_MARKETS_ENABLED` defaults to empty → `TARGET_MARKETS = ['global']` → same Phase 1 behavior. The fanout ONLY activates when the operator sets the env var (post-Starter-upgrade). No code-level guard needed beyond the default; the operator-setup section above is informational for the SUMMARY.

Purpose: Deliver SOCIAL-02, SOCIAL-03, MARKET-01, MARKET-02. Phase 2's core scraper change. After this plan, the data pipeline supports per-market scraping; plans 02-04 and 02-05 layer the dashboard UI on top.

Output: A refactored `scrapers/apify_social.py` that runs identical to Phase 1 when the flag is off, and fans across all 8 APAC markets when the flag is on. Plus extended tests proving both modes.
</objective>
